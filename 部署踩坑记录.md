# 部署踩坑记录

> 服务器：腾讯云 Ubuntu Server 22.04.5 LTS
> 时间：2026-07-10
> 项目：AI Agent Platform (FastAPI + LangGraph + Qwen + PostgreSQL/pgvector + Redis)

---

## 问题 1：Kubernetes apt 源 DNS 失效

**现象**

```text
E: The repository 'https://apt.kubernetes.io kubernetes-xenial Release' does not have a Release file.
```

`apt-get update` 直接报错，导致 `get.docker.com` 安装脚本无法执行。

**原因**

服务器上残留了过时的 Kubernetes apt 源配置，该源地址已失效。

**解决**

```bash
sudo rm -f /etc/apt/sources.list.d/kubernetes.list
sudo rm -f /etc/apt/sources.list.d/apt.kubernetes.io.list
sudo sed -i '/kubernetes/s/^/#/' /etc/apt/sources.list
```

删除冲突源后重新 `apt-get update` 即可。

## 问题 2：腾讯云内网 apt 镜像 DNS 解析失败

**现象**

```text
W: Failed to fetch http://mirrors.tencentyun.com/ubuntu/dists/jammy/InRelease
  Something wicked happened resolving 'mirrors.tencentyun.com:http' (-5 - No address associated with hostname)
```

清理 Kubernetes 源后，`apt-get update` 仍然失败。

**原因**

腾讯云 Ubuntu 镜像默认配置了内网 apt 源 `mirrors.tencentyun.com`，但该域名的 DNS 解析在该实例上不生效（可能 VPC 网络配置问题）。

**解决**

将 `/etc/apt/sources.list` 中的腾讯内网镜像替换为阿里云公网镜像：

```bash
sudo sed -i 's|mirrors.tencentyun.com|mirrors.aliyun.com|g' /etc/apt/sources.list
sudo apt-get update
```

## 问题 3：Docker Hub 镜像加速器失效

**现象**

```text
ERROR: failed to do request: Head "https://hub-mirror.c.163.com/v2/library/nginx/manifests/alpine"
  dial tcp: lookup hub-mirror.c.163.com on 223.5.5.5:53: no such host
```

`docker compose build` 时无法拉取 `python:3.11-slim`、`node:22-alpine`、`nginx:alpine` 等基础镜像。

**原因**

服务器 Docker 的 `daemon.json` 配置了已失效的镜像加速器（`hub-mirror.c.163.com`、`mirror.baidubce.com`），这些加速器已停止服务或域名无法解析。

**解决**

1. 测试可用的镜像加速器：

```bash
nslookup docker.m.daocloud.io   # 解析成功
```

2. 用远程 Python 写入正确的 daemon.json（避免 shell 转义破坏 JSON）：

```bash
python3 -c "import json; f=open('/tmp/daemon.json','w'); json.dump({'registry-mirrors':['https://docker.m.daocloud.io']}, f, indent=2); f.close()"
sudo cp /tmp/daemon.json /etc/docker/daemon.json
sudo systemctl restart docker
```

3. 验证镜像源生效：

```bash
sudo docker info | grep -A2 "Registry Mirrors"
# 应显示: https://docker.m.daocloud.io/
```

> ⚠️ **坑**：不要用 `echo '{"key":"value"}' | sudo tee` 写 JSON，shell 转义会破坏双引号导致 JSON 格式无效。Docker 日志会报 `invalid character 'r' looking for beginning of object key string`。用 Python 写最可靠。

## 问题 4：前端 TypeScript 类型错误

**现象**

```text
src/components/MessageList.tsx(58,9): error TS2322: Type 'unknown' is not assignable to type 'ReactNode'.
```

`docker compose build frontend` 在 `tsc -b && vite build` 阶段失败。

**原因**

`message.metadata?.toolCall` 的类型是 `unknown`（来自 `[key: string]: unknown` 索引签名），直接用于 JSX 条件渲染时 `unknown | JSX.Element` 联合类型不满足 `ReactNode` 约束。

**解决**

```tsx
// 修改前 — toolCall 是 unknown，&& 表达式类型不合法
{message.metadata?.toolCall && (
  <div>...</div>
)}

// 修改后 — 显式类型检查，条件表达式返回 boolean
{typeof message.metadata?.toolCall === "string" && message.metadata.toolCall.length > 0 && (
  <div>...</div>
)}
```

## 问题 5：dashscope 包版本不存在

**现象**

```text
ERROR: No matching distribution found for dashscope==1.22.3
```

`pip install` 在容器内失败。

**原因**

`requirements.txt` 中指定的 `dashscope==1.22.3` 在 PyPI 上不存在（版本号跳过了，从 1.22.2 直接到 1.23.0）。

**解决**

项目通过 OpenAI 兼容接口调用 Qwen，实际不依赖 `dashscope` SDK。直接移除该依赖：

```text
# requirements.txt 修改
openai>=1.50.0
anthropic>=0.40.0
# dashscope uses OpenAI-compatible endpoint, no native SDK needed
```

## 问题 6：LangChain 包版本冲突

**现象**

```text
ERROR: Cannot install -r requirements.txt (line 17), (line 19), (line 20) and langchain-core==0.3.28
  because these package versions have conflicting dependencies.
ERROR: ResolutionImpossible
```

**原因**

`langgraph==0.2.68`、`langchain-openai==0.2.14`、`langchain-community==0.3.14` 各自对 `langchain-core` 有不同的版本要求，精确锁定版本导致 pip 无法找到同时满足所有包的 `langchain-core` 版本。

**解决**

将 LangChain 系列包的版本锁定从 `==` 改为 `>=`，让 pip 自动解析兼容版本：

```text
# 修改前
langgraph==0.2.68
langchain-core==0.3.28
langchain-openai==0.2.14
langchain-community==0.3.14

# 修改后
langgraph>=0.2.60
langchain-core>=0.3.20
langchain-openai>=0.2.10
langchain-community>=0.3.10
```

## 问题 7：SQLAlchemy 关系映射缺失

**现象**

```text
sqlalchemy.exc.InvalidRequestError: Mapper 'Mapper[Conversation(conversations)]' has no property 'agent'.
  If this property was indicated from other mappers or configure events, ensure registry.configure() has been called.
```

后端启动即崩溃，uvicorn worker 进程退出。

**原因**

`AgentConfig` 模型定义了：

```python
conversations: Mapped[list["Conversation"]] = relationship(back_populates="agent")
```

`back_populates="agent"` 要求 `Conversation` 模型也有一个名为 `agent` 的 relationship，但 `Conversation` 模型只定义了外键 `agent_id`，没有定义对应的 relationship。

**解决**

在 `Conversation` 模型中补全 relationship：

```python
class Conversation(Base):
    ...
    agent_id: Mapped[Optional[str]] = mapped_column(ForeignKey("agent_configs.id"))
    agent: Mapped[Optional["AgentConfig"]] = relationship(back_populates="conversations")
```

## 问题 8：Dockerfile 内 apt-get 下载极慢

**现象**

后端镜像构建卡在 `RUN apt-get update && apt-get install` 步骤超过 20 分钟无响应。

**原因**

`python:3.11-slim` 基础镜像内的 apt 源是 `deb.debian.org`，从国内服务器访问极慢。

**解决**

在 Dockerfile 的 apt 命令前替换为阿里云 Debian 镜像：

```dockerfile
RUN sed -i 's|deb.debian.org|mirrors.aliyun.com|g' /etc/apt/sources.list.d/debian.sources 2>/dev/null || \
    sed -i 's|deb.debian.org|mirrors.aliyun.com|g' /etc/apt/sources.list 2>/dev/null || true

RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential libpq-dev && rm -rf /var/lib/apt/lists/*
```

> **注意**：Debian Trixie（python:3.11-slim 基于 Trixie）的源配置文件在 `/etc/apt/sources.list.d/debian.sources`，不是传统的 `/etc/apt/sources.list`。需要两个 `sed` 兼容。

## 问题 9：Dockerfile 内 pip install 慢

**现象**

apt 步骤加速后，`pip install` 仍然较慢，偶尔超时。

**原因**

默认 pip 源 `pypi.org` 从国内访问不稳定。

**解决**

在 `pip install` 命令中直接指定阿里云 PyPI 镜像：

```dockerfile
RUN pip install --no-cache-dir \
    -i https://mirrors.aliyun.com/pypi/simple/ \
    --trusted-host mirrors.aliyun.com \
    -r requirements.txt
```

> 不要用 `pip config set` 再 install 的方式（两步可能在不同 RUN 层之间状态不一致），直接在 `pip install` 命令加 `-i` 参数最可靠。

## 总结

| # | 问题 | 根因 | 解决方式 |
|---|------|------|----------|
| 1 | Kubernetes apt 源报错 | 旧源残留 | 删除 sources.list.d 下的 kubernetes 源 |
| 2 | 腾讯云内网镜像 DNS 失败 | mirrors.tencentyun.com 不解析 | 替换为 mirrors.aliyun.com |
| 3 | Docker Hub 镜像拉取失败 | 旧加速器域名失效 | 配置 daocloud 镜像源 |
| 4 | 前端 TS 类型错误 | unknown 类型用于 JSX | 显式 typeof 检查 |
| 5 | dashscope 版本不存在 | PyPI 无此版本号 | 移除（用 OpenAI 兼容接口） |
| 6 | LangChain 版本冲突 | 精确版本互相不兼容 | 改用 >= 范围约束 |
| 7 | SQLAlchemy 关系缺失 | back_populates 未双向定义 | 补全 Conversation.agent relationship |
| 8 | Dockerfile apt 慢 | Debian 官方源国内慢 | 替换为阿里云 Debian 镜像 |
| 9 | Dockerfile pip 慢 | PyPI 官方源国内慢 | pip install 加 -i 阿里云源 |

### 经验教训

1. **国内服务器部署，第一件事就是换镜像源** — apt、pip、Docker Hub 全部换国内镜像
2. **daemon.json 用 Python 写，不用 echo** — shell 转义会破坏 JSON 双引号
3. **Python 包不要锁死精确版本** — 用 `>=` 让 pip 自动解决依赖冲突
4. **SQLAlchemy relationship 必须双向定义** — `back_populates` 两端都要有对应属性
5. **TypeScript strict 模式下 unknown 类型不能直接用于 JSX** — 需要显式类型收窄
