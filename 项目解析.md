# AI Agent 平台 — 面试项目深度解析

> 本文档面向面试场景，从技术选型、架构设计、难点攻克三个维度对项目进行拆解。

---

## 一、项目概览

**AI Agent Platform** 是一个面向生产的 AI Agent SaaS 应用，核心能力包括：

- **流式对话**：支持多轮对话，SSE 实时推送 token
- **RAG 知识库**：基于 pgvector 的语义检索增强生成
- **工具调用（Function Calling）**：内置 9 个工具（计算器、搜索、天气、汇率等）
- **多 LLM 切换**：Qwen（默认）、OpenAI、Claude，一套接口统一调度
- **Agent 可配置**：系统提示词、温度、最大 token、启用工具均可在线调整

**技术栈一句话总结：** React + FastAPI + LangGraph + PostgreSQL/pgvector + Redis + Docker Compose，全异步架构，面向国内网络环境深度优化。

## 二、技术选型 & 为什么不用其他方案

### 2.1 后端框架：FastAPI

| 维度 | FastAPI | Flask | Django |
|------|---------|-------|--------|
| 异步原生支持 | ✅ `async/await` 一等公民 | ❌ 需插件（`flask[async]`） | ⚠️ 4.1+ 部分异步 |
| 性能 | 高（Starlette + Uvicorn，接近 Node.js 水平） | 中（WSGI，同步阻塞） | 中（默认同步） |
| 自动 API 文档 | ✅ Swagger UI + ReDoc 开箱即用 | ❌ 需 flask-restx 等扩展 | ✅ DRF 自带 |
| 类型安全 | ✅ Pydantic 模型驱动，请求/响应自动校验 | ❌ 手动校验 | ⚠️ DRF Serializer |
| SSE 流式支持 | ✅ StreamingResponse 原生支持 | ⚠️ 需手动处理 | ⚠️ 需 Channels |
| 学习曲线 | 低 | 最低 | 高 |

**为什么选 FastAPI？**

1. **Agent 场景天然需要异步** — LLM 调用是 I/O 密集型，同步阻塞会严重浪费资源。FastAPI 的 `async def` 端点配合 `StreamingResponse` 可以同时处理大量 SSE 长连接。
2. **Pydantic 与 LangChain 生态一致** — LangChain 和 LangGraph 大量使用 Pydantic 做数据模型，FastAPI 同样基于 Pydantic，类型体系天然统一。
3. **SSE 流式传输是核心功能** — FastAPI 的 `StreamingResponse` + `sse-starlette` 实现 SSE 比 Flask 少很多样板代码。

**为什么不选 Django？**
- Django 太重，自带 ORM、模板引擎、Admin 等我们用不上。Agent 项目需要轻量灵活，Django 的"全家桶"反而增加认知负担。
- Django REST Framework 的 Serializer 不如 Pydantic 类型严格，且异步支持滞后。

**FastAPI 的缺点：**
- 生态不如 Django 成熟（没有 Admin、没有成熟的权限系统）
- 大型项目缺乏社区最佳实践（目录结构、分层建议各执一词）

### 2.2 Agent 框架：LangGraph

| 维度 | LangGraph | 自实现循环 | CrewAI | AutoGPT |
|------|-----------|-----------|--------|---------|
| 图状态管理 | ✅ 内建 StateGraph + 自动追加消息 | ❌ 需手动管理 | ⚠️ 基于角色，非通用图 | ⚠️ 绑定特定范式 |
| 流式输出 | ✅ `stream_mode="messages"` | ❌ 需手动实现 | ⚠️ 有限 | ⚠️ 有限 |
| 条件路由 | ✅ `add_conditional_edges` | ❌ 需手写 if-else | ❌ 固定流程 | ❌ 固定流程 |
| 工具绑定 | ✅ 与 LangChain Tool 无缝集成 | ⚠️ 手动绑定 | ✅ | ✅ |
| 可观测性 | ✅ LangSmith 一键接入 | ❌ | ❌ | ❌ |
| 灵活性 | 极高（自定义节点 + 边） | 极高 | 中 | 低 |

**为什么选 LangGraph？**

1. **图结构天然匹配 Agent 推理循环**：`LLM 思考 → 决定调用工具 → 工具执行 → LLM 再思考`，这就是一个有向环状图。LangGraph 的 `StateGraph` 可以精确建模这个流程。
2. **流式支持到位**：`stream_mode="messages"` 可以逐 token 输出 LLM 响应，这正是聊天应用最核心的体验。自己实现需要处理 chunk 解析、增量拼接等脏活。
3. **安全机制内置**：通过迭代计数器 `iteration` 防止无限循环（本项目限制 10 轮），LangGraph 的条件路由天然支持这种防护。
4. **可观测性即生产力**：接入 LangSmith 后可以在 UI 中回放每一次 Agent 执行的完整链路（RAG 检索→LLM 调用→工具调用→循环），调试效率极大提升。

**为什么不选 CrewAI / AutoGPT？**
- CrewAI 强在多 Agent 协作（角色扮演），单 Agent + 工具调用的场景下反而过度抽象。
- AutoGPT 绑定特定范式（计划-执行-验证），不符合我们"对话式 Agent"的灵活交互需求。
- 自己实现 Agent 循环虽然可控性最高，但需要处理状态管理、流式输出、错误恢复等大量工程细节，投入产出比低。

**LangGraph 的缺点：**
- 学习曲线陡峭：需要理解 StateGraph、Node、Edge、Command、Checkpointer 等概念
- LangChain 版本兼容问题多发：不同包之间依赖关系复杂，升级困难（本项目踩过坑）
- 文档不够成熟：很多高级用法找不到示例，需要读源码

### 2.3 LLM 选择：Qwen 作为默认

| 维度 | Qwen (DashScope) | OpenAI | Claude |
|------|-------------------|--------|--------|
| 国内访问 | ✅ 阿里云，低延迟 | ❌ 需科学上网 / 代理 | ❌ 需科学上网 / 代理 |
| 中文能力 | ⭐⭐⭐⭐⭐ 顶级 | ⭐⭐⭐⭐ 良好 | ⭐⭐⭐⭐ 良好 |
| 价格 | ¥0.0008/1K tokens | $0.0025/1K tokens | $0.003/1K tokens |
| API 兼容性 | ✅ OpenAI 兼容接口 | ✅ 原生 | ⚠️ 需桥接 |
| 多模态 | ✅ qwen-vl-plus | ✅ GPT-4V | ✅ Claude 3 Vision |
| 数据安全 | ✅ 数据不出境 | ⚠️ 境外处理 | ⚠️ 境外处理 |

**为什么默认 Qwen？**

1. **国内部署刚需**：项目面向国内服务器部署，阿里云 DashScope 无需代理，延迟 < 100ms，而 OpenAI 可能 TLS 握手上秒。
2. **中文能力拉满**：Qwen 系列模型在中文问答、文档理解方面明显优于同级别的 GPT/Claude 模型。
3. **OpenAI 兼容接口**：DashScope 提供标准的 `/v1/chat/completions` 接口，可以用 OpenAI SDK 直接调用，切换成本几乎为零。
4. **价格优势明显**：成本约为 GPT-4o 的 1/10，业务可持续性更高。

**为什么不只支持一个提供商？**
- 可靠性：单一提供商有宕机风险；支持三个提供商意味着随时可以切换。
- 灵活性：不同任务适合不同模型（代码→Claude，长文→Qwen-Max，性价比→Qwen-Turbo）。
- 面向未来：通过 LLM 工厂模式 + 缓存，新增提供商只需加一个 case 分支。

### 2.4 向量数据库：pgvector + PostgreSQL

| 维度 | pgvector | Milvus | Pinecone | Weaviate |
|------|----------|--------|----------|----------|
| 部署复杂度 | 低（PostgreSQL 扩展） | 高（独立服务） | 低（SaaS） | 中（自建或云） |
| 运维成本 | 极低（复用 PG） | 高 | 零（按量付费） | 中 |
| 与业务数据关联 | ✅ JOIN 查询 | ❌ 独立存储 | ❌ 独立存储 | ⚠️ 有限 |
| 性能 | 中（百万级向量） | 高（十亿级） | 高 | 高 |
| 费用 | 免费 | 免费（自建） | 按量付费 | 免费版有限 |
| ACID 事务 | ✅ PG 原生支持 | ❌ | ❌ | ❌ |

**为什么选 pgvector？**

1. **架构极简**：一个 PostgreSQL 实例同时处理业务数据（对话、消息、Agent 配置）和向量检索（文档分块），无需额外部署向量数据库服务。对中小型项目，少一个服务就是少一个故障点。
2. **数据一致性**：`Document` 和 `DocumentChunk` 在同一个数据库中，可以用 `JOIN` 查询、享受 ACID 事务保护。假设用户删除一个文档，所有关联的 chunk 通过 `CASCADE` 自动清理，不会有孤儿数据。
3. **运维友好**：DBA 不需要学习新的向量数据库运维，pgvector 就是 PostgreSQL 的一个扩展。
4. **成本为零**：不需要为 Pinecone 等 SaaS 服务付费。

**为什么不选 Milvus / Pinecone？**
- Milvus：功能强大但运维复杂（需要 etcd + MinIO + Pulsar / Kafka），对 2-4GB 内存的小项目不划算。
- Pinecone：SaaS 服务，数据出境风险 + 持续付费，国内网络延迟也可能较高。

**pgvector 的缺点：**
- 百万级以上向量的检索性能不如专用向量数据库（但小项目够用）
- 索引构建（IVFFlat / HNSW）需要调参，默认参数不够优化
- 嵌入维度（本项目 1024）受 PostgreSQL 行大小限制

### 2.5 缓存 & 限流：Redis

| 维度 | Redis | Memcached | 内存字典 |
|------|-------|-----------|---------|
| 持久化 | ✅ RDB + AOF | ❌ 纯内存 | ❌ 进程重启丢失 |
| 数据结构 | 丰富（String/Hash/List/Set/SortedSet） | 仅 String | 基本 |
| 原子操作 | ✅ INCR、EXPIRE 等 | ⚠️ 有限 | ❌ 需加锁 |
| 多进程共享 | ✅ | ✅ | ❌ 单进程 |

**为什么选 Redis？**

1. **速率限制需要原子性**：本项目用 `INCR` + `EXPIRE` 实现滑动窗口限流，这是 Redis 的经典用法，内存字典无法保证多 Worker（4 个 Uvicorn）间的原子性。
2. **会话缓存**：`cache_set` / `cache_get` 可以缓存 Agent 配置、对话摘要等热数据，减少数据库查询。
3. **Docker 部署极简**：`redis:7-alpine` 镜像仅 ~30MB，启动 1 秒。

**Redis 的缺点：**
- 引入额外服务（虽然轻量），增加部署复杂度
- 内存占用随数据增长，需要配置 `maxmemory` 和淘汰策略

**为什么选 React + Vite + Tailwind？**

1. **React**：生态最成熟，处理流式数据更新（SSE 逐 token 追加消息）有丰富的 hooks 支持，且招聘市场需求最大（面试时这是加分项）。
2. **Vite**：冷启动秒级、HMR 即时生效，开发体验远优于 Webpack。本项目没有复杂的 Webpack 插件依赖，Vite 完全够用。
3. **Tailwind CSS**：没有用任何 UI 组件库（Ant Design、MUI），而是基于 Tailwind 手写了整个 Cyberpunk/Neon 主题的 UI 系统。这样做的理由：
   - **差异化设计**：Ant Design 的企业蓝白风格千篇一律，AI 产品需要更有辨识度的 UI
   - **打包体积可控**：Tailwind 按需生成 CSS，最终样式文件远小于 component library
   - **不受框架约束**：升级、定制、改造完全自主可控

**为什么不选 Next.js？**
- 本项目是纯 SPA，没有 SEO 需求，不需要 SSR/SSG。
- Agent 聊天应用前端逻辑简单（单一聊天视图），Next.js 的路由、服务端组件等特性用不上，反而增加复杂度。
- Vite 开发体验（尤其是 HMR 速度）优于 Next.js。

### 2.7 部署：Docker Compose + GitHub Actions + Nginx

**为什么 Docker Compose 而不是 K8s？**

1. **项目规模匹配**：4 个服务（前端、后端、PostgreSQL、Redis）用 Docker Compose 是黄金配比。K8s 需要至少 3 台 Master + 若干 Node，资源开销和运维成本完全不成比例。
2. **单机部署场景**：早期用户量不大，单机 4GB 内存即可运行全部服务，无需水平扩展。
3. **配置简洁**：一个 `docker-compose.yml` 文件就描述清楚所有服务的依赖关系、端口映射、环境变量，新人 5 分钟就能理解。

**Nginx 的设计考量：**

- 前端容器内置 Nginx，承担两个角色：（1）静态文件服务（React SPA），（2）反向代理（`/api/*` → `backend:8000`）
- SSE 流式传输的关键配置：`proxy_buffering off`、`proxy_cache off`、`proxy_read_timeout 300s`，这三个配置缺一不可，否则 SSE 连接会被 Nginx 缓冲或超时断开。

## 三、核心架构设计

### 3.1 整体数据流

```
用户输入 "今天天气怎么样"
    ↓
[React] sendMessage() → fetch POST /api/chat (SSE)
    ↓
[Nginx] 反向代理 → backend:8000
    ↓
[FastAPI] Chat 端点
    ├─ deps.py: verify_rate_limit() → Redis
    ├─ deps.py: get_or_create_conversation() → PostgreSQL
    ├─ deps.py: get_default_agent() → PostgreSQL
    ↓
[LangGraph] run_agent() 异步生成器
    ├─ [RAG] rag_node() → pgvector 语义搜索
    ├─ yeld "rag_context" 事件
    ├─ [LLM] agent_node() → Qwen DashScope API
    │   └─ 系统提示词 + RAG上下文 + 用户消息 → LLM
    ├─ [Router] should_continue()
    │   ├─ 有 tool_call → ToolNode(tool名称)
    │   │   ├─ yield "tool_start" 事件
    │   │   └─ 返回 agent_node 再思考
    │   └─ 无 tool_call → END
    ├─ yield "token" 事件（逐 token）
    └─ yield "done" + 来源信息
    ↓
[FastAPI] StreamingResponse → SSE text/event-stream
    ↓
[React] useChat → 异步生成器逐条消费事件
    ├─ "rag_context" → 显示来源引用
    ├─ "token" → 追加到助手消息
    ├─ "tool_start" → 显示工具调用提示
    └─ "done" → 完成，显示最终消息
```

### 3.2 LLM 工厂模式

```python
# backend/app/agent/llm.py — 核心设计

_llm_cache: dict[str, BaseChatModel] = {}

def get_llm(provider, model, temperature, max_tokens) -> BaseChatModel:
    cache_key = f"{provider}_{model}_{temperature}_{max_tokens}"
    if cache_key not in _llm_cache:
        if provider == "qwen":
            _llm_cache[cache_key] = ChatOpenAI(
                base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
                api_key=settings.DASHSCOPE_API_KEY,
                model=model,
                streaming=True,  # 关键：始终启用流式
            )
        elif provider == "openai":
            _llm_cache[cache_key] = ChatOpenAI(model=model, ...)
        elif provider == "claude":
            _llm_cache[cache_key] = ChatOpenAI(
                base_url="https://api.anthropic.com/v1/",
                ...
            )
    return _llm_cache[cache_key]
```

**设计亮点：**

1. **统一接口**：三个提供商全部适配为 LangChain 的 `BaseChatModel`，上层调用代码无差异
2. **实例缓存**：相同配置的 LLM 实例复用，避免重复创建连接
3. **OpenAI 兼容桥接**：Claude 通过 Anthropic 的 OpenAI 兼容端点接入，用同一套 SDK

### 3.3 Graph 缓存

```python
# backend/app/agent/graph.py

_graph_cache: dict[str, StateGraph] = {}

def get_agent_graph(enabled_tools: list[str]) -> StateGraph:
    cache_key = ",".join(sorted(enabled_tools))
    if cache_key not in _graph_cache:
        graph = StateGraph(AgentState)
        graph.add_node("agent", agent_node)
        graph.add_node("tools", create_tool_node(enabled_tools))
        graph.add_edge(START, "agent")
        graph.add_conditional_edges("agent", should_continue, {
            "tools": "tools",
            "end": END,
        })
        graph.add_edge("tools", "agent")
        _graph_cache[cache_key] = graph.compile()
    return _graph_cache[cache_key]
```

**为什么缓存 Graph？**
- LangGraph 的 `compile()` 会做图验证、拓扑排序、边检查，有计算开销。
- 工具配置相同的请求可以复用同一个编译后的 Graph。
- 缓存键是"启用的工具列表排序后拼接"，保证相同配置命中缓存。

### 3.4 RAG 设计

**为什么不把 RAG 做成 LangChain Tool？**

做成 Tool 的话：
```
LLM 收到用户问题 → 决定要不要调用 rag_search → 调用 → 拿到结果 → 再思考 → 回复
```
问题是：**LLM 不一定每次都想到调用 RAG**，即使知识库中有相关文档也可能被跳过。

本项目的做法：
```
用户消息 → 系统前置检索（rag_node）→ 将结果注入系统提示词 → LLM 直接基于上下文回答
```

**好处：**
- RAG 检索是确定性的，100% 执行，不依赖 LLM 的判断
- 减少一轮 LLM 往返（省 token + 省延迟）
- 系统提示词中的上下文比工具返回的上下文更容易被模型"遵从"

## 四、踩坑 & 难点解决

### 4.1 LangChain 版本地狱

**问题：** 用 `==` 精确锁定版本后，不同包之间的传递依赖互相冲突。例如 `langchain-core==0.2.x` 和 `langgraph==0.2.x` 要求不同版本的 `pydantic`。

**解决：** 将所有依赖改为 `>=` 约束，让 pip 的依赖解析器自动计算兼容的版本组合。代价是每次 `pip install` 稍慢，但避免了版本冲突。`requirements.txt` 中的每个包都经过验证在当前组合下能正常工作。

### 4.2 国内网络环境适配（系列问题）

| 问题 | 影响 | 解决方案 |
|------|------|---------|
| Dockerfile `apt update` 连接 Debian 官方源超时 | 构建失败 | 替换为阿里云镜像 `mirrors.aliyun.com` |
| Dockerfile `pip install` 连接 PyPI 超时 | 构建失败 | 添加 `-i https://mirrors.aliyun.com/pypi/simple/` |
| Docker Hub 拉取镜像缓慢 | 部署 30 分钟+ | 配置 `docker.m.daocloud.io` 镜像加速器 |
| `git pull` 拉取 GitHub 代码 TLS 中断 | CI/CD 不可用 | 改为 tar + SCP：GitHub Runner 打包 → SSH 传输 |
| 腾讯云服务器残留 K8s apt 源 DNS 失效 | `apt update` 失败 | 删除 `/etc/apt/sources.list.d/` 过期源 |
| dashscope SDK 版本号在 PyPI 不存在 | `pip install` 失败 | 放弃 SDK，改用 OpenAI 兼容接口 |

### 4.3 Docker 环境变量覆盖问题（502 Bad Gateway）

**问题：** 本地开发时 `.env` 中 `POSTGRES_HOST=localhost`，Docker 部署时应该是 `POSTGRES_HOST=postgres`（服务名）。直接使用同一个 `.env` 导致线上 502。

**解决：** 不在 `.env` 中改动，而是在 `docker-compose.yml` 中用 `environment` 覆盖这三个变量：
```yaml
backend:
  environment:
    - POSTGRES_HOST=postgres
    - REDIS_HOST=redis
    - APP_HOST=0.0.0.0
```
`environment` 优先级高于 `env_file`，本地开发不受影响，Docker 环境自动覆盖。

### 4.4 SSE 流式传输的数据库会话管理

**问题：** SSE 是一个长时间运行的生成器。FastAPI 的 `get_db()` 依赖注入会在请求返回后清理数据库会话，但 SSE 流可能在请求函数返回后还在向生成器内部写入数据，导致 `AsyncSession` 被提前关闭。

**解决：** 在 SSE 生成器内部创建独立的数据库会话：
```python
async def event_generator():
    async with async_session_factory() as db:
        # 在这个上下文管理器内做所有数据库操作
        ...
    # 会话在这里安全关闭
```
`finally` 块中保存助手消息，确保即使 Agent 执行异常，已经生成的对话内容也不会丢失。

### 4.5 前端 TypeScript 严格模式下的类型收窄

**问题：** SSE 事件中 `data` 字段是 `unknown` 类型，直接用在 JSX 中报错 `Type 'unknown' is not assignable to type 'ReactNode'`。

**解决：** 在 JSX 中使用前添加显式的 `typeof` 检查：
```typescript
{typeof event.data === 'string' ? event.data : JSON.stringify(event.data)}
```

### 4.6 Agent 无限循环防护

**问题：** LLM 可能陷入"调用工具 → 拿到结果 → 再调用同一个工具"的死循环。

**解决：** 在 `AgentState` 中设置 `iteration` 计数器，每次 LLM 思考 +1，达到 10 次后强制结束。LangGraph 的条件路由中增加迭代次数判断：
```python
def should_continue(state: AgentState):
    if state["iteration"] >= 10:
        return "end"
    last_msg = state["messages"][-1]
    if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
        return "tools"
    return "end"
```

### 4.7 多模态图片支持的模型切换

**问题：** 用户上传图片时需要 vision 模型（如 `qwen-vl-plus`），但普通文本对话用 vision 模型浪费 token 和费用。

**解决：** LLM 工厂函数检测消息中是否包含 `image_url`，自动切换到配置的 vision 模型：
```python
def get_llm(provider, model, ..., message_content=None):
    if has_image_content(message_content):
        model = get_vision_model(provider)  # qwen-vl-plus
    ...
```

### 4.8 DuckDuckGo 搜索在国内不可用

**问题：** DuckDuckGo Instant Answer API 对中文搜索返回空结果。

**解决：** 改用 `duckduckgo_search` Python 库的 HTML 抓取模式（绕过 API），同时对搜索结果做中文语言过滤。

## 五、项目亮点（面试时重点讲）

1. **全异步架构**：从 FastAPI 异步端点 → SQLAlchemy 异步 ORM → Redis 异步客户端 → LangGraph 异步流式输出，全链路异步避免了 Python GIL 的性能瓶颈，4 个 Worker 即可承载数百并发 SSE 长连接。

2. **巧妙的 Graph 缓存设计**：相同工具配置的 LangGraph 实例缓存复用，避免了重复编译开销和潜在的内存泄漏。

3. **RAG 检索前置化**：不把 RAG 做成 LLM 可选调用的工具，而是系统级前置检索直接注入上下文，保证检索确定性，减少一次 LLM 往返。

4. **国内环境深度适配**：从 apt 源、pip 源、Docker 镜像加速、CI/CD 架构（免 GitHub 直连）、LLM 默认选择，全部针对国内服务器和网络环境做了深度优化。每一个踩过的坑都有完整的记录和解决方案。

5. **多 LLM 提供商统一接口**：LLM 工厂 + 实例缓存 + OpenAI 兼容桥接，三个提供商用一套代码接入，新增提供商只需加一个分支。

6. **SSE 流式工程细节**：独立数据库会话管理避免连接泄漏、`finally` 块保证异常时消息不丢失、Nginx SSE 配置（buffering off）、逐 token 推送 + 工具调用事件分离。

7. **Cyberpunk 主题 UI**：没有用第三方 UI 库，基于 Tailwind CSS 手写完整的 Neon/Cyber 风格设计系统，包括发光阴影、毛玻璃面板、动画网格背景、自定义滚动条等，差异化设计让产品有辨识度。

8. **完整的 CI/CD**：GitHub Actions → tar + SCP（绕墙）→ 服务器自动构建 + 部署 + 健康检查，代码推送后 5 分钟内自动上线。
