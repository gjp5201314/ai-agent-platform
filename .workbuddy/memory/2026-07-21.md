## 2026-07-21 — Mock 模式实现

### 新增功能：Mock 模式
用户可在前台一键启用/关闭 Mock 模式，启用后所有对话返回模拟响应，无需配置任何 LLM API Key。

### 修改文件清单

**后端 (6 个文件):**
1. `backend/app/config.py` — 新增 `mock_mode_enabled: bool = False` 全局配置
2. `backend/app/schemas.py` — ChatRequest 新增 `mock_mode` 字段；新增 `MockModeStatus`、`MockModeToggle` schema
3. `backend/app/agent/mock_agent.py` **(新建)** — Mock Agent 核心逻辑，支持：
   - 4 种预设回复模板（greeting/code/knowledge/default），根据用户消息自动分类
   - 流式 token 输出（按自然边界分块，模拟网络延迟）
   - 可选工具调用模拟（calculator）
   - 完整的 SSE 事件序列（conversation_id 由 chat.py 统一处理）
4. `backend/app/api/chat.py` — 流式和非流式路径均支持 mock 路由：
   - `_prepare_chat` 正常执行（DB 操作仍真实）
   - `_stream_response` 根据 `mock_mode` 标志选择 `run_mock_agent` 或 `run_agent`
   - mock 优先级：per-request flag > global setting
5. `backend/app/api/admin.py` — 新增 2 个管理 API：
   - `POST /api/v1/admin/mock/status` — 查询 mock 状态
   - `POST /api/v1/admin/mock/toggle` — 切换全局 mock 模式
6. `.env.example` — 新增 `MOCK_MODE_ENABLED` 配置项说明

**前端 (5 个文件):**
1. `frontend/src/api/client.ts` — `streamChat` 新增 `mockMode` 参数；新增 `mockStatus`、`mockToggle` API
2. `frontend/src/hooks/useChat.ts` — `sendMessage` 新增 `mockMode` 参数透传
3. `frontend/src/App.tsx` — 新增 `mockMode` 状态管理，传递到 ChatInterface
4. `frontend/src/components/ChatInterface.tsx` — 新增 mock 模式黄色横幅提示；传递 props
5. `frontend/src/components/MessageInput.tsx` — 新增 Mock 模式切换按钮（FlaskConical 图标，琥珀色主题），与 RAG 按钮并列

### 架构设计
- Mock 路由在 `_stream_response` 层完成，`_prepare_chat` 始终执行（DB 操作保持真实）
- Mock 模式可全局开启（环境变量 `MOCK_MODE_ENABLED=true`）或在前台按请求开启
- 前台 UI 有双重视觉提示：顶部横幅 + 输入区 badge + 按钮高亮

## 2026-07-21 — 本地启动环境修复

### 用户反馈：双击 `start-dev.bat` 窗口闪退

经过多轮排查，最终定位 4 个独立问题，**全部修复**：

### 问题 1：bat 文件 UTF-8 编码被 GBK 解析
- **症状**：双击后窗口瞬间关闭，看不到任何错误
- **根因**：bat 用 Write 工具保存为 UTF-8，但 Windows cmd.exe 中文版默认按 GBK 解析，路径里的中文字符（`项目`、`甘俊培`）全乱码
- **修复**：bat 改纯 ASCII + 用 `%~dp0` 和 `%USERPROFILE%` 替代硬编码中文路径

### 问题 2：Python venv 缺依赖
- **症状**：`ModuleNotFoundError: No module named 'loguru'`
- **根因**：venv 里没装 `requirements.txt`
- **修复**：执行 `pip install -r backend/requirements.txt` 装齐

### 问题 3：Docker 不在 PATH
- **症状**：`'docker' 不是内部或外部命令`
- **根因**：双击 bat 启动的新 cmd 没有 Docker 安装目录
- **修复**：bat 开头 `set PATH=C:\Program Files\Docker\Docker\resources\bin;%PATH%`

### 问题 4：`start` 命令嵌套引号断裂
- **症状**：后端窗口打开后立即关闭
- **根因**：`cmd /k "cd /d "%PATH%" && "..."` 内层引号提前闭合外层
- **修复**：改用 `start /D "%PATH%"` 设工作目录，避开嵌套引号

### 问题 5：`pause >nul` 隐藏提示
- **症状**：用户以为窗口关闭
- **根因**：`pause >nul` 吞掉 "Press any key..." 提示
- **修复**：改正常 `pause`

### 问题 6：前端 `node_modules` 缺 `react-router-dom`
- **症状**：浏览器报 "Failed to resolve import 'react-router-dom'"
- **根因**：`npm install` 没装完整
- **修复**：`cd frontend && npm install`

### 问题 7：后端启动后所有 API 报 500
- **症状**：前端能调通但全部 500
- **根因 A**：`column agent_configs.allow_delegation does not exist`
  - 模型有 `allow_delegation` 字段但数据库旧表没这列
  - **修复**：在 `backend/app/database.py` 的 `init_db` 中加幂等迁移
- **根因 B**：`redis.exceptions.AuthenticationError: HELLO must be called with the client already authenticated`
  - `redis-py` 5.x 默认 RESP3，与密码保护的 Redis 7 不兼容
  - **修复**：`backend/app/core/redis_client.py` 加 `protocol=2` 强制 RESP2
- **根因 C**：`backend/.env` 的 `REDIS_PASSWORD=` 是空的
  - 但 docker-compose 启 Redis 时用了默认密码 `redis_secure_password`
  - **修复**：在 `.env` 设 `REDIS_PASSWORD=redis_secure_password`

### 经验总结（重要）
1. **Windows bat 文件必须用纯 ASCII** 或在 Windows 系统自带编辑器中保存为 GBK，不能用 UTF-8
2. **bat 里的所有路径必须用环境变量**（`%~dp0`, `%USERPROFILE%`）避免编码问题
3. **`start /D` 是设子进程工作目录的正确方式**，避免 `cd && command` 的嵌套引号
4. **`redis-py` 5.x + Redis 7 + 密码 = 必须设 `protocol=2`**，这是已知坑
5. **`.env` 必须与 `docker-compose.yml` 配置一致**，尤其密码
6. 首次启动项目至少要：`pip install` + `npm install` + 数据库迁移

### 修改的文件
- `start-dev.bat`（重写）
- `backend/app/database.py`（加 `allow_delegation` 迁移）
- `backend/app/core/redis_client.py`（加 `protocol=2`）
- `backend/.env`（设 `REDIS_PASSWORD`；`QWEN_MODEL` → `qwen3.7-plus`）
- `.env.example`（同步默认值）

## 2026-07-21 — LLM 错误兜底处理

### 问题
Qwen `qwen-plus` 免费额度耗尽（AllocationQuota.FreeTierOnly），所有对话请求卡死不返回任何内容，用户只能看到加载动画。

### 根因
`agent_node` 中 `llm.ainvoke()` 抛出的异常被 LangGraph 内部吞掉，`graph.astream()` 不是正常 raise 而是静默终止流，导致 SSE 流在 `conversation_id` 后卡住，不发送 error 也不发送 done。

### 修复：3 层防御
1. **`nodes.py` → `agent_node`**：`llm.ainvoke()` 包裹 `try/except`，按错误类型（quota/auth/timeout/rate/model_not_found/other）返回中文友好提示
2. **`graph.py` → `run_agent`**：`graph.astream()` 包裹 `try/except Exception`，LangGraph 吞掉的异常在这里兜底，yield error token
3. **`chat.py` → `_stream_response`**（已有，无改动）：第三层安全网，但已不再需要

### 效果对比
- **修复前**：请求 → `conversation_id` → 永远卡死（无任何错误信息）
- **修复后**：请求 → 立即返回友好提示，告诉用户原因 + 3 种解决办法 + 技术细节

### 友好提示示例

```
2. 切换到 Mock 模式继续测试（点击输入框旁的 Mock 按钮）
3. 在管理后台切换到其他可用的模型
```
